Improve the Deal Builder UI and layout without changing the working business logic. The current functionality is working. This task is focused on UI clarity, layout cleanup, and conditional field visibility so the page supports fast, confident execution during a phone call.

GOAL:
Make the Deal Builder feel like a clean review-and-execute workflow instead of a cluttered form. Preserve all existing functionality, pricing logic, execution logic, and integrations.

--------------------------------------------------
1. OVERALL UX GOAL
--------------------------------------------------

The page should support this user flow:

1. Identify the customer
2. Confirm deal type and delivery / fulfillment method
3. Review live spot and products
4. Review summary
5. Acknowledge terms
6. Execute the trade

Visually, the page should feel lighter, cleaner, and easier to scan. The user should immediately understand what matters most and what action happens next.

Do not redesign the application into a new concept. Improve the current Deal Builder layout and presentation using the existing structure and functionality.

--------------------------------------------------
2. CUSTOMER + BILLING ADDRESS MUST BE COMBINED
--------------------------------------------------

The current layout separates customer information at the top and billing address much farther down the page. That is poor UX and should be fixed.

Required change:
- Move Billing Address into the same top card/section as Customer
- Customer and Billing Address should be the first thing addressed on the page
- They should feel like one logical unit

Recommended structure inside a single combined card:
- Customer section
  - First Name
  - Last Name
  - Email
  - Phone
  - State
  - ZIP
  - Lead ID
  - Confirmation ID
- Billing Address section
  - Street Address
  - Apt / Suite
  - City
  - State
  - ZIP

Rules:
- Keep this as one combined top-left card or one clearly unified panel
- Preserve existing field bindings and data handling
- Do not break lead prefill behavior
- Do not move Billing Address to the bottom again

--------------------------------------------------
3. DEAL TYPE / DELIVERY VISIBILITY RULES
--------------------------------------------------

The page must adapt based on the selected Deal Type.

A. If Deal Type = CASH
- Show the Delivery section
- Delivery options should remain available:
  - FedEx Hold
  - Home Delivery

B. If Deal Type = IRA
- Hide the Delivery section entirely
- Do not show FedEx Hold
- Do not show Home Delivery
- Do not show manual delivery address fields
- Do not show FedEx location selection or hours

Important:
- Eventually IRA deals will use depository information instead of delivery
- For now, when IRA is selected, simply hide the delivery UI cleanly
- Do not show placeholder delivery controls for IRA
- Do not remove the ability to add depository information later

Implementation guidance:
- Structure this so the Delivery section can later be replaced or extended with a Depository section for IRA deals
- Do not hardwire the UI in a way that makes future IRA depository support difficult

--------------------------------------------------
4. DELIVERY SECTION CLEANUP FOR CASH DEALS
--------------------------------------------------

For CASH deals, improve the Delivery section so it is not redundant or confusing.

A. FedEx Hold selected
Show only:
- selected FedEx location card
- location hours
- any fields actually required for execution

Hide:
- redundant manual delivery entry fields if they are not required for FedEx Hold
- any duplicate location/address presentation that creates confusion

B. Home Delivery selected
Show only:
- the delivery address fields needed for Home Delivery

Hide:
- FedEx Hold location card
- FedEx Hold hours
- irrelevant FedEx Hold address details

Goal:
The user should only see the inputs relevant to the currently selected delivery mode.

RIGHT COLUMN
