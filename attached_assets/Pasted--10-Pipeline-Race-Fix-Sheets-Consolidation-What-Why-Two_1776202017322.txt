#10 - Pipeline Race Fix & Sheets Consolidation

## What & Why

Two bugs need to be fixed together because they both sit in the same Sheets write layer.

### 1. Pipeline race condition

Right now, `syncProspectToPipeline` is called from both `leads.ts` and `scheduling.ts`.

In normal flow, the lead route should own initial prospect creation. The scheduling route should only merge appointment details into that existing prospect row.

Because both routes currently write to the Prospecting Pipeline, a user who submits a lead form and books immediately can produce duplicate rows with the same Lead ID.

### 2. Sheets consolidation

`writeDealToBuilderSheet` and `appendDealToOpsSheet` currently depend on separate spreadsheet IDs (`GOOGLE_DEAL_BUILDER_SHEET_ID`, `GOOGLE_DEALS_OPS_SHEET_ID`).

Those env vars were never set in Railway, so those functions skip deal writes.

We are consolidating all tabs into the single master spreadsheet:
`1BYnY_VSCxz_Ec9hkLBZOobz4bzDD2vkPmyMloPAHzsE`

That means:

* Prospecting Pipeline
* booking_attempts
* Deal Builder
* Deals
* Operations

all live in one spreadsheet and should use the same `SPREADSHEET_ID`.

## Done looks like

* Submitting a lead form and immediately booking an appointment produces exactly one row in Prospecting Pipeline
* That row is updated with scheduling fields
* Locking a deal writes to the Deal Builder tab
* Locking a deal writes to the Deals tab
* If current logic also writes to Operations in this phase, confirm that explicitly
* No code references `GOOGLE_DEAL_BUILDER_SHEET_ID` or `GOOGLE_DEALS_OPS_SHEET_ID`
* Railway startup no longer warns about those env vars
* `writeDealLinkToMasterSheet` continues to update Deal ID + Open Deal link correctly

## Out of scope

* Moving tabs in Google Sheets manually
* Removing old Railway env vars manually
* Recap email / invoice PDF
* Deal Builder UI changes
* Deal lock API response changes

## Architecture constraints

* Normal business flow is lead-first
* `leads.ts` owns initial prospect creation
* `scheduling.ts` must not call `syncProspectToPipeline`
* `scheduling.ts` should only call `mergeAppointmentIntoPipeline`

`mergeAppointmentIntoPipeline` must:

1. try to find the existing row by Lead ID
2. update scheduling fields if found
3. if not found, retry once after a short delay
4. if still not found and fallback lead data is present, insert a full row as a defensive fallback only

Retry is a resilience layer only, not the primary integrity mechanism.

The scheduling route must pass all lead/body fields needed for defensive fallback insertion:

* firstName
* lastName
* email
* phone
* state
* allocationType
* allocationRange
* timeline
* formType
* createdAt
* leadId

`DEAL_BUILDER_SHEET_ID` and `DEALS_OPS_SHEET_ID` module constants must be deleted.

`writeDealToBuilderSheet` and `appendDealToOpsSheet` must use `SPREADSHEET_ID`.

Any early return guards based on the removed env vars must also be deleted.
