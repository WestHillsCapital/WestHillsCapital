## 1. Purpose

This policy defines the types of logs generated by the organization's systems, where they are stored, and how long they are retained. Security-relevant logs are retained for a minimum of 12 months to support incident investigation, SOC 2 audit evidence, and regulatory compliance.

---

## 2. Scope

All logs generated by the Docuplete platform, the West Hills Capital internal portal, and all supporting infrastructure — including application logs, database audit records, error tracking events, and access logs.

---

## 3. Log types, retention windows, and storage locations

### 3.1 Application logs (pino / Railway)

| Property | Value |
|---|---|
| **Description** | Structured JSON logs emitted by the Express API server and BullMQ worker via the pino logger. Include request handling, scheduler ticks, job processing, and INFO/WARN/ERROR-level operational events. |
| **Storage location** | Railway log storage (cloud-managed); aggregated view available in the Railway dashboard |
| **Minimum retention** | **12 months** |
| **Enforcement** | Railway's log retention window must be configured to retain at least 12 months. If Railway's default is shorter, export logs to a long-term sink (e.g., external log aggregation service) before they expire. |
| **PII risk** | May contain email addresses in error context (e.g., `actorEmail` in log fields). Do not export to unauthorized third parties. |

### 3.2 Error tracking events (Sentry)

| Property | Value |
|---|---|
| **Description** | JavaScript/TypeScript exceptions, unhandled promise rejections, and manually captured errors. Captured via `@sentry/node` in `artifacts/api-server/src/instrument.ts`. |
| **Storage location** | Sentry cloud (organization: `joe-airesearch`, project: `node-express`) |
| **Minimum retention** | **90 days** (Sentry plan default; upgrade to 12 months if available on current plan) |
| **PII risk** | Stack traces may contain partial PII. Sentry's data scrubbing is enabled. Do not export raw events to unauthorized recipients. |
| **Action required** | Verify Sentry plan retention settings annually during policy review. |

### 3.3 Audit log — security events (`org_audit_log`)

| Property | Value |
|---|---|
| **Description** | Structured security event records written for every security-sensitive user action: API key create/revoke, 2FA enable/disable, session revocation, trusted device revocation, team membership changes, plan changes, data exports, and deletion requests. Written to the PostgreSQL `org_audit_log` table via `insertAuditLog()`. Critical events re-throw on write failure to prevent silent omission. |
| **Storage location** | PostgreSQL (production database) |
| **Minimum retention** | **12 months** |
| **Enforcement** | No automated pruning exists for `org_audit_log`. Rows are never deleted. If database storage becomes a concern, archive rows older than 12 months to cold storage rather than deleting them. |
| **Schema reference** | `artifacts/api-server/src/lib/auditLog.ts`, `artifacts/api-server/src/db.ts` |

### 3.4 PDF audit events (`pdf_audit_events`)

| Property | Value |
|---|---|
| **Description** | Access events for generated PDF documents: who downloaded a PDF, from which IP, at what time. Written by `recordPdfAuditEvent()` in `artifacts/api-server/src/lib/pdf-audit.ts`. |
| **Storage location** | PostgreSQL `pdf_audit_events` table |
| **Minimum retention** | **12 months** |
| **Enforcement** | No automated pruning. Archive rows older than 12 months if storage becomes a concern. |

### 3.5 Login history (`user_login_history`)

| Property | Value |
|---|---|
| **Description** | Records of successful user logins: Clerk user ID, IP address, user agent, and timestamp. Used for security review and incident investigation. |
| **Storage location** | PostgreSQL `user_login_history` table |
| **Minimum retention** | **90 days** |
| **Enforcement** | Automated pruning via the `prune:session-data` BullMQ job, which runs every 24 hours and deletes rows older than 90 days. |

### 3.6 Active session records (`user_active_sessions`)

| Property | Value |
|---|---|
| **Description** | Records of active user sessions, used to show users their device list and to support session revocation. |
| **Storage location** | PostgreSQL `user_active_sessions` table |
| **Minimum retention** | **30 days** (after which inactive sessions are pruned) |
| **Enforcement** | Automated pruning via the `prune:session-data` BullMQ job (30-day window). |

### 3.7 Webhook delivery logs (`webhook_deliveries`)

| Property | Value |
|---|---|
| **Description** | Record of every outbound webhook attempt: event type, HTTP status, response body, delivery duration, and attempt number. Used for debugging and auditing outbound data flows. |
| **Storage location** | PostgreSQL `webhook_deliveries` table |
| **Minimum retention** | **90 days** |
| **Enforcement** | No automated pruning currently. The Engineering Lead should review and implement a pruning job if table growth becomes significant. |

### 3.8 Booking attempts (`booking_attempts`)

| Property | Value |
|---|---|
| **Description** | Records of appointment booking attempts (success and failure), including IP address, email, and error codes. Used for rate limiting and fraud detection. |
| **Storage location** | PostgreSQL `booking_attempts` table |
| **Minimum retention** | **90 days** |
| **Enforcement** | Automated pruning via the `prune:audit-tables` BullMQ job, which runs every 24 hours and deletes rows older than 90 days. |

### 3.9 Interview session data (`docufill_interview_sessions`)

| Property | Value |
|---|---|
| **Description** | Submitted customer interview sessions including encrypted answers, signer identity, and generated PDF references. Retention is configurable per Docuplete account. |
| **Storage location** | PostgreSQL `docufill_interview_sessions` table |
| **Minimum retention** | Per account's configured `submission_retention_days` setting; default is no automatic deletion |
| **Maximum retention** | Accounts may configure automatic deletion; the `prune:submissions` job enforces this daily |
| **Enforcement** | `prune:submissions` BullMQ job runs every 24 hours. |
| **Note** | This is customer-controlled data. The organization must honor deletion requests in accordance with applicable data protection laws. |

## 4. Minimum security log retention summary

The following log types are considered **security-relevant** and must be retained for a minimum of **12 months**:

| Log type | Storage | Current retention |
|---|---|---|
| Application logs (Railway/pino) | Railway log storage | Configure to ≥ 12 months |
| Security audit log (`org_audit_log`) | PostgreSQL | Indefinite (no automated deletion) |
| PDF audit events (`pdf_audit_events`) | PostgreSQL | Indefinite (no automated deletion) |
| Sentry error events | Sentry cloud | Verify plan; target ≥ 12 months |

---

## 5. Log integrity

Audit log entries in `org_audit_log` and `pdf_audit_events` are append-only in normal operation. No update or delete queries are issued against these tables by application code. Any modification to these records outside the application should be treated as a potential security incident and investigated per the [Incident Response Plan](../incident-response-plan.md).

---

## 6. Log access control

- Railway logs: accessible to Railway project team members only (see [Access Control Policy](access-control-policy.md))
- Sentry events: accessible to Sentry organization members only
- PostgreSQL audit tables: accessible via `DATABASE_URL` — production DB access is restricted per the Access Control Policy

---

## 7. Log disposal

When logs reach the end of their retention window and are deleted:

- Automated deletion (BullMQ jobs) is performed via SQL `DELETE` queries that overwrite the row data as part of normal PostgreSQL vacuuming.
- Railway log exports, if any exist, should be securely deleted from the export destination.
- There is no requirement to cryptographically erase individual log records unless they contain Restricted-tier data (see [Data Classification Policy](data-classification-policy.md)).

---

## 8. Policy review

This policy is reviewed annually by the Engineering Lead. The retention windows and enforcement mechanisms in Section 3 are verified against the current BullMQ job configuration and database schema at each review. Any gaps in automated enforcement are documented and assigned for remediation.
